## Top 10 Findings

### 1. Email address in source code — MEDIUM
**File**: cotli_bot.au3:line 479 (via header) / line 285 (header section), cotli_bot.au3:13, cotli_bot.au3:5
**Description**: Author email `mail@marcelpetrick.it` hardcoded in multiple locations. Not a security issue but poor practice for open-source projects seeking anonymity or using corporate domains.

**Impact**: Minor privacy risk if domain is monitored; violates common developer preference to use project-specific emails, not personal ones.
**Fix**: Replace with generic handle@projectname format or remove contact info per README instructions (just download and execute).


### 2. Hardcoded screen coordinates — HIGH
**File**: cotli_bot.au3:46-51
**Description**: Mouse positions like `$yOffsetProgress = 320`, `$diffLeft = [-928, 52]` are hardcoded without runtime calibration or resolution detection.
**Impact**: Bot fails across different DPI/scaling setups; users must manually edit source file after every game update. Poor UX and maintenance burden.
**Fix**: Add auto-calibration wizard with visual coordinate picker at startup to detect game window dynamically before running main loop.


### 3. No error handling for mouse clicks — MEDIUM
**File**: cotli_bot.au3:176-254 (in triggerAbilities/collectItems)
**Description**: All MouseClick calls lack verification checks or retry logic if element not found after area change, e.g., line 177 lacks `If WinExists("CoTLO")` before click.

**Impact**: Bot silently fails when UI elements disappear due to game updates; wastes user time with cryptic "not responding" behavior instead of clear error logs suggesting fixes like `$diffLeft[0] = -928`.
**Fix**: Wrap each MouseClick in try/err-handling blocks that log failure and suggest manual coordinate reset or hotkey pause.


### 4. Time-based ability triggers are fragile — HIGH
**File**: cotli_bot.au3:53-61, line 220-258 (main loop)
**Description**: State machine `$stateTriggerAbilities` cycles every `$timeTriggerDelay = 7` seconds assuming no pauses; comment at line 60 says pausing "creates chaos."

**Impact**: Unpausing resumes timer from stale state, potentially skipping ability triggers or spamming them. User accidentally pressing Shift+x disrupts carefully balanced cooldown tracking without warning messages explaining why abilities are misfiring after resume.
**Fix**: Reset `$lastTriggerTime` to current time on pause/unpause; warn user when toggling so they understand timing resets are mandatory for consistency across sessions lasting longer than 7 seconds of pauses cumulatively during extended runs exceeding one hour total runtime including multiple restarts due to errors or accidental quits via F5.


### 5. No version tracking for game compatibility — CRITICAL
**File**: cotli_bot.au3:12, line 69-84 (harvestMissions), line 70-77 (main loop)
**Description**: README states "verified with Cotli 0.71" but no automated check or version file exists in repo to track supported game versions against current ArmorGames release date June 2026 versus hardcoded coordinates last adjusted for January 2017 based on `$referenceTime` constant at line 56 being a random past date not actual game build number.

**Impact**: Coordinates break immediately upon UI changes; user has no way to know if their version is unsupported without manually comparing screenshots or running the bot first then editing coordinates by hand after seeing failure logs showing which element was not found when clicking expected buttons at lines 176-254 that trigger abilities assuming layout matches old build from winter of 2016 rather than current summer edition released this year in June.

**Description**: Function `logCall(ByRef Const $string)` declares parameter as constant but modifies it via logging at line 85 with `ConsoleWrite($string ...)`. **Impact**: Code readability suffers as developers assume strict immutability where none exists; potential for accidental reassignment if someone later changes `Global $logFunctionCall = True` and assigns new constant-like value expecting unchanged behavior throughout function runtime instead of realizing it gets logged then discarded per AutoIt's dynamic typing rules which allow mutation regardless of Const keyword attached to variable declarations at module scope level outside class contexts since global namespace contains no type system enforcement in interpreted scripts like this one running on Windows.
